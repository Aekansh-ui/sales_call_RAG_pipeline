**What happened:**
- **T-01.4 — docker-compose.yml:**
  - Taught the concept: what Docker Compose is and why it replaces `docker run` chains
  - Taught the service graph with ASCII diagram showing postgres, ollama, app and how they connect
  - Taught 3 key concepts in depth:
    1. **Internal DNS** — service name = hostname inside Docker network; `postgres:5432` not `localhost:5432`
    2. **`depends_on` with `condition: service_healthy`** — waits for healthcheck to pass, not just container start
    3. **Named volumes** — data persists across `docker compose down`; wiped only with `-v` flag
  - Built the file in 3 passes (postgres → ollama → app):

  **Pass 1 — postgres service:**
  - `image: pgvector/pgvector:pg16` — pre-built Postgres 16 + pgvector, no manual extension compile
  - `env_file: .env` + `environment:` — loads .env then maps specific keys
  - `postgres_data:/var/lib/postgresql/data` — named volume for DB persistence
  - `ports: "5432:5432"` — host:container format; for host access only (containers use internal DNS)
  - `healthcheck: pg_isready` — makes depends_on actually wait for Postgres to be ready

  **Pass 2 — ollama service:**
  - `image: ollama/ollama:latest`
  - `ollama_data:/root/.ollama` — models (~2.3 GB total) persist here, not re-downloaded on restart
  - `healthcheck: curl /api/tags` — confirms ollama server is accepting requests
  - No `env_file` needed — ollama is self-contained

  **Pass 3 — app service:**
  - `build: .` — builds from our Dockerfile, not a pre-built image
  - `volumes: - ./:/app` — bind mount: host code is instantly visible inside container (no rebuild on code change)
  - `ports: "${FASTAPI_PORT:-8000}:8000"` — `:-8000` is a default fallback if .env var missing
  - `depends_on: condition: service_healthy` — waits for BOTH postgres and ollama to be healthy before starting

### Prompt 3
> "yes" (approving T-01.5)

**What happened:**
- **T-01.5 — Postgres pgvector init script:**
  - Taught the `/docker-entrypoint-initdb.d/` mechanism:
    - Official Postgres Docker image scans this directory on **first boot only** (empty volume)
    - Runs every `.sql` and `.sh` file in alphabetical order as superuser
    - On every subsequent restart, Postgres finds data in the volume and skips this directory entirely
    - Why not run it manually? Reproducibility — `docker compose up` should produce a fully working setup with zero manual steps
  - Created `docker/init-pgvector.sql`:
    ```sql
    CREATE EXTENSION IF NOT EXISTS vector;
    ```
  - Mounted it into the postgres container in docker-compose.yml:
    ```yaml
    - ./docker/init-pgvector.sql:/docker-entrypoint-initdb.d/init-pgvector.sql:ro
    ```
  - `:ro` = read-only mount — container can read but not modify the source file
  - Taught the **only-runs-once gotcha**: to re-trigger init scripts, you must `docker compose down -v` (destroy volume) then `docker compose up` — this will be `make reset-db` in T-01.7

  Validated:
  ```bash

  # → VALID
  ```
