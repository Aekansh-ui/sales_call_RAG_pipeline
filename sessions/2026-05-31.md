### Prompt 10
> "done, after i ran the command it has installed the pip packages and lot of exporting. explain what we did here."

**What happened:**
Explained the full `docker build` sequence step by step:
- Step 1: `FROM` — Docker pulled `python:3.11-slim` from Docker Hub (big download, first time only)
- Step 2: `ENV` — set two env vars, instant
- Step 3: `RUN apt-get` — installed ffmpeg + libgomp1 inside the container; `rm -rf /var/lib/apt/lists/*` in same command to avoid bloating the layer
- Step 4: `COPY requirements.txt .` — only requirements.txt, not the rest of the code (cache isolation)
- Step 5: `RUN pip install` — the "exporting" the user saw; pip downloads wheels from PyPI, unpacks, installs to site-packages; `--no-cache-dir` saves ~100-200 MB
- Step 6: `COPY . .` — entire project directory copied into `/app`
- Step 7: `EXPOSE` — metadata only
- Key concept reinforced: image = frozen read-only snapshot; container = live running instance from that image

### Prompt 11
> "what if i want to change the requirements.txt file in future, how do i handle caching in that case?"

**What happened:**
- Explained that Docker hashes every `COPY`'d file — if `requirements.txt` hash changes, that layer and all layers below it are invalidated (cache miss)
- Docker automatically re-runs `pip install` when `requirements.txt` changes — no manual intervention needed
- Taught the stale container gotcha: a running container won't pick up the new image automatically
- Correct workflow: `docker compose down` then `docker compose up --build -d` (the `--build` flag forces image rebuild)
- Taught the 3-row mental model: code change (~5 sec) vs requirements change (1-3 min) vs Dockerfile change (varies)
