### Key decisions & why

- **UUID PKs** — generated in Python before insert, safe across environments, no collision risk
- **`TIMESTAMPTZ` not `TIMESTAMP`** — always stores UTC, avoids timezone bugs
- **`ON DELETE CASCADE`** — delete a call → all chunks/embeddings/topics/sentiment auto-deleted
- **`UNIQUE (call_id, chunk_index)`** — chunk index is only unique within a call, not globally
- **`vector(768)`** — nomic-embed-text output dimension
- **IVFFlat index, `lists=50`, `vector_cosine_ops`** — cosine because meaning is in direction not magnitude; IVFFlat for ANN search (faster than brute force at scale)
- **Migration 001 has no `vector` column** — pgvector extension must be enabled first (migration 002); migrations build on each other in order

## Concepts taught this session

1. **Relational schema design & normalization** — store each fact once, reference via FK
2. **UUID vs SERIAL PKs** — UUID safer for distributed generation; SERIAL leaks row count
3. **`TIMESTAMPTZ` vs `TIMESTAMP`** — always use TIMESTAMPTZ for created_at
4. **`ON DELETE CASCADE`** — automatic child-row cleanup when parent deleted
5. **What a `vector(768)` column is** — 768 floats, output of nomic-embed-text
6. **pgvector operators** — `<->` L2, `<#>` inner product, `<=>` cosine
7. **Why cosine over L2 for embeddings** — meaning encoded in direction, not magnitude
8. **IVFFlat internals** — k-means clusters at build time; probes buckets at query time; tradeoff: recall vs speed
9. **Connection pooling** — keeps connections open, reuses them; `ThreadedConnectionPool(minconn=1, maxconn=5)`
10. **Migrations as version control for DB** — numbered files run in order, idempotent
